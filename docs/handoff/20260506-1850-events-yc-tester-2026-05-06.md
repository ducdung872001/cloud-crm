## BE-1. Tenant portal settings — đồng bộ cross-browser

### Hiện trạng FE
Tại `/crm/ch_events` → "Cài đặt portal sự kiện" — admin sửa banner, tagline,
tone màu… cho trang public `/crm/events`. **Lưu cục bộ trên `localStorage`**
theo tenant (key `reborn.community_hub.portal_settings.<bsnId>`). Code:
`src/pages/CommunityHub/Events/portalSettings.ts`.

### Vấn đề
Admin sửa setting trên máy A → mở trên máy B (cùng tenant) thì không thấy.
Tester yêu cầu sync qua BE để cross-machine cùng admin/cùng tenant.

### Đề xuất API
```
GET    /market/tenant-settings?key=community_hub_portal
PUT    /market/tenant-settings   { key: "community_hub_portal", value: <JSON> }
```

- Resolve tenant theo header `Hostname` (FE đã gửi qua `fetchApi`) hoặc theo
  `Authorization` token.
- Value là JSON tự do, ~2-5KB. BE chỉ store + serve — không validate schema.
- Ưu tiên RW cho user role admin; ai cũng được GET (cần cho user public
  thấy banner).

### Ghi chú FE
Khi BE ready, FE chỉ cần sửa `portalSettings.ts` để gọi API trước khi đọc
LS. Luồng hiện tại đã có FE comment: *"Lưu cục bộ trên trình duyệt theo
tenant — khi BE hỗ trợ tenant settings sẽ đồng bộ tự động"*.

## BE-2. Slot count desync khi đăng ký

### Tester repro
- Mở 2 chrome cùng event `gim-cn-adn` (cùng tenant W-HOUSE, bsnId=6).
- Chrome A: đăng ký 1 user → thành công.
- Chrome A vẫn thấy `Còn 100/100 chỗ` (không trừ).
- Chrome B (admin tab "Người đăng ký"): thấy `Còn 99/100 chỗ` (đã trừ).

Screenshot: `docs/requirements/20260506/bug.docx` image7 + image8.

### Phân tích
- FE đọc `event.maxAttendees` + `event.activeRegistrations` (từ stats endpoint).
- Có lẽ BE chỉ trừ slot khi reg.status = `confirmed`, nhưng đăng ký mới
  status = `pending` → không trừ.
- Hoặc cache layer FE/CDN đang serve stale.

### Yêu cầu
1. Đảm bảo slot trừ ngay khi `pending` (nếu policy "giữ chỗ chờ duyệt") —
   hoặc rõ ràng "chỉ trừ khi confirmed" và FE đổi label thành
   `Còn N/M chỗ (chưa kể M đang chờ duyệt)`.
2. `GET /market/events/public/get?slug=...` phải trả `activeRegistrations`
   chính xác, không cache stale > 1s.
3. Confirm với FE policy nào áp dụng để FE update label.

## BE-4. Comments API cho events

### Hiện trạng FE
`src/pages/CommunityHub/Events/components/EventComments.tsx` lưu comments
**hoàn toàn trên localStorage** theo browser. Hệ quả:
- User A bình luận trên chrome → user B mở trên firefox không thấy.
- **Admin cũng không thấy comment của user** (mỗi browser có LS riêng).

Tester báo cụ thể: `/crm/events/gim-cn-adn` — bình luận người khác và admin
không thấy được.

### Đề xuất API
```
GET  /market/events/{eventId}/comments?includeHidden=false
POST /market/events/{eventId}/comments    { authorName, authorPhone?, authorMemberCode?, authorRole, parentId?, content }
PATCH /market/events/comments/{id}        { isHidden?, status? }   // admin only
```

Schema (xem `src/pages/CommunityHub/Events/types.ts` `EventComment`):
```ts
{
  id: string
  eventId: string
  parentId?: string         // reply 1 cấp
  authorName: string
  authorPhone?: string
  authorMemberCode?: string
  authorRole: "guest" | "member" | "admin" | "moderator"
  content: string
  status?: "approved" | "pending" | "rejected"   // moderation
  isHidden?: boolean
  hiddenReason?: string
  createdAt: ISO
  updatedAt?: ISO
  reviewedAt?: ISO
  reviewedBy?: string
}
```

### Yêu cầu
1. CRUD endpoint trên + tenant-scoped theo Hostname/token.
2. POST không cần auth (guest comment được). Admin verify role qua token.
3. Pagination bằng cursor (eventId, createdAt) — events lớn có thể >100 cmts.
4. Trigger notification (optional): khi user comment → email admin BTC.

### FE swap
Khi BE ready, FE refactor `eventCommentsStorage` ở `EventComments.tsx`
(hiện localStorage-only) sang gọi API + giữ LS làm cache offline. Ước
tính 1-2h FE.
